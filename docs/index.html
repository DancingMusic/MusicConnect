DancingMusic Connector Ecosystem

MusicConnect Reference (Open-Design)

This site is the official documentation and development center for the DancingMusic/MusicConnect connector SDK. Independent connector implementation repositories are loaded separately from the host app at runtime. The recommended plugin flow is:

Host app

Loads connectors by URL, normalizes config schema, and renders player/store flows.

Connector repo

Implements MusicConnector and publishes dist/index.js.

GitHub Pages

Publishes API docs and demo pages for preview and onboarding.

Overview

MusicConnect owns only music source contract and base adapters. It does not contain player logic or visual rendering. Every official connector follows a single SemVer release process and can be upgraded in the host via pinned CDN URLs.

Connector Contract

Implementations MUST provide these interfaces:

  • meta ({ id / name / version / capabilities / ... })
  • search(query, options) for keyword queries
  • getTrack(trackId) for track detail
  • getStreamUrl(trackId) for playable stream
  • getLyrics(trackId) (if available)
  • init(config?) for runtime config injection

Any runtime-specific behavior (DOM manipulation, canvas draw, queue mutation) must stay in host or plugin layers.

Quick Start

Note Local build and contract tests should pass before any release. 
import { MusicStoreClient } from "@dancingmusic/music-store";
import { NeteaseConnector } from "@dancingmusic/music-connect/netease";

const connector = new NeteaseConnector();
await connector.init({ apiBaseUrl: "https://netease-cloud-music-api.vercel.app" });

const client = new MusicStoreClient();
await client.connectors.register(connector);
const active = client.connectors.active?.meta;
// publishable dist for host loading

Docs + GitHub Pages

For any connector repo, follow the same page pattern: left nav + scrollable article + inline end TOC. This repo also publishes docs here to provide a central entry for the connector specification.

  1. Drop your final docs into docs/index.html.
  2. Keep the article readable in direct GitHub Pages rendering.
  3. Keep links to demo, config and API sections stable for host UI onboarding.

Release Workflow

Release process is tag-driven from connector repos:

  • Update version in package.json
  • Run typecheck/build/test
  • Commit and push to main
  • Create release tag (vX.Y.Z) and GitHub Release
  • Host uses pinned tag URLs, never floating @main
